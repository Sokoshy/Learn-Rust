### Similarités entre RefCell&lt;T>/Rc&lt;T> et Mutex&lt;T>/Arc&lt;T>

Vous avez peut-être remarqué que `counter` est immuable mais que nous pouvons obtenir une référence mutable à la valeur qu'il contient ; cela signifie que `Mutex<T>` fournit une mutabilité intérieure, comme le fait la famille `Cell`. De même que nous avons utilisé `RefCell<T>` dans le Chapitre 15 pour nous permettre de muter le contenu à l'intérieur d'un `Rc<T>`, nous utilisons `Mutex<T>` pour muter le contenu à l'intérieur d'un `Arc<T>`.

Un autre point à noter est que Rust ne peut pas vous protéger de tous les types d'erreurs logiques lorsque vous utilisez `Mutex<T>`. Rappelez-vous dans le Chapitre 15 que l'utilisation de `Rc<T>` comportait le risque de créer des cycles de références, où deux valeurs `Rc<T>` se réfèrent mutuellement, provoquant des fuites de mémoire. De même, `Mutex<T>` comporte le risque de créer des _deadlocks_. Ceux-ci se produisent lorsqu'une opération nécessite de verrouiller deux ressources et que deux threads ont chacun acquis l'un des verrous, les faisant attendre l'un pour l'autre indéfiniment. Si vous êtes intéressé par les deadlocks, essayez de créer un programme Rust qui a un deadlock ; ensuite, recherchez des stratégies d'atténuation des deadlocks pour les mutex dans n'importe quel langage et essayez de les implémenter en Rust. La documentation de l'API de la bibliothèque standard pour `Mutex<T>` et `MutexGuard` offre des informations utiles.

Vous pouvez également lire la [section du livre sur les traits `Send` et `Sync`](https://doc.rust-lang.org/stable/book/ch16-04-extensible-concurrency-sync-and-send.html) et voir comment nous pouvons les utiliser avec des types personnalisés.
